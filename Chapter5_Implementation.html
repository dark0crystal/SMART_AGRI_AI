Chapter 5: Project Implementation

This chapter describes how the VerdeX design presented in Chapter 4 was translated into working software. The implementation spans three major subsystems: the Machine Learning (ML) inference pipeline built on the EfficientNet-B1 architecture, the Django REST Framework backend serving as the system's intelligence layer, and the Flutter mobile frontend providing a bilingual cross-platform interface for Omani farmers. A dedicated data access layer mediates all reads and writes against Google Cloud Firestore. The chapter is organised into three sections: Section 5.1 documents key coding decisions for each component; Section 5.2 describes the verification strategy applied to each module; and Section 5.3 presents the validation approach used to confirm the system meets its stated requirements.

5.1   Coding

The implementation adhered closely to the layered architecture defined in the design phase. Figure 5.1 provides a component-level overview of the deployed system. Each component was developed and tested in isolation before integration, consistent with the iterative Agile methodology adopted by the team.

Figure 5.1 – VerdeX System Component Diagram

5.1.1   Machine Learning Module

The ML module, located in the backend/vision/ package, implements the image-based disease classification pipeline. It was designed to be loosely coupled from the Django application layer: all PyTorch and timm imports are performed lazily, allowing the backend to start and serve text-only requests even when GPU libraries are unavailable on the host.

5.1.1.1   Model Architecture and Checkpoint Loading

The image classifier uses an EfficientNet-B1 backbone loaded through the timm library [1]. EfficientNet-B1 was selected because it offers a larger receptive field compared to the EfficientNetB0 described in the proposal, while remaining computationally affordable for CPU inference during the testing phase. The checkpoint stores the model's state dictionary together with a class_names list and training configuration (model name, dropout rate), enabling the application to reconstruct the exact architecture at inference time without hard-coding hyperparameters in the codebase.

Table 5.1 – Checkpoint Dictionary Keys

KeyTypeDescription
class_namesList[str]Ordered list of 8 disease class labels matching softmax output indices
model_state_dictOrderedDictPyTorch parameter tensors for the full EfficientNet-B1 + classification head
config / argsdictTraining hyperparameters: model name, dropout (default 0.3), drop_path_rate (0.2)

The checkpoint loading function (vision/checkpoint.py: load_checkpoint) reconstructs the model using timm.create_model with pretrained=False, then loads the saved weights with strict=True to ensure no parameter mismatch is silently ignored. The model is set to evaluation mode (model.eval()) immediately after loading to disable dropout and batch-normalisation training behaviour. A global singleton (_vision_bundle) caches the loaded model, class names, and torch device so that the expensive checkpoint deserialisation occurs only once per process lifecycle, regardless of how many concurrent diagnosis requests are handled.

# vision/checkpoint.py (simplified) def load_checkpoint(model_path, device, *, default_model_name="efficientnet_b1"): checkpoint = torch.load(model_path, map_location=device, weights_only=False) class_names = checkpoint["class_names"] # e.g. ["Anthracnose", ...] model_name = checkpoint.get("config", {}).get("model") or default_model_name dropout = checkpoint.get("config", {}).get("dropout", 0.3) model = timm.create_model(model_name, pretrained=False, num_classes=len(class_names), drop_rate=dropout, drop_path_rate=0.2) model.load_state_dict(checkpoint["model_state_dict"], strict=True) model.to(device).eval() return model, class_names

5.1.1.2   Inference Pipeline

The inference function predict_from_pil (vision/predict.py) accepts a PIL Image object and returns the predicted class label, top-1 confidence score, and a ranked top-5 list. The pre-processing transform consists of three steps: (i) resize to 224×224 pixels using bilinear interpolation, (ii) convert to a float tensor with values in [0, 1], and (iii) normalise channel-wise using the ImageNet mean ([0.485, 0.456, 0.406]) and standard deviation ([0.229, 0.224, 0.225]). These statistics match the pre-training distribution of the EfficientNet-B1 backbone, preserving the discriminative features learned during transfer learning. Inference is wrapped in torch.no_grad() to suppress gradient computation and reduce memory consumption. The raw logits are passed through torch.softmax, and torch.topk extracts the five highest-scoring classes.

5.1.1.3   Witch's Broom False-Positive Guard

During iterative expert evaluation, the team identified a systematic false-positive pattern: the Witch's Broom class was predicted with an uncertain margin over the true disease in images containing yellowing foliage—a symptom shared by multiple conditions. A post-inference guard was implemented in ai_service.py to mitigate this. The rule states that if (a) the top-1 predicted label is Witch's Broom, and (b) the probability gap between the top-1 and top-2 predictions is at or below the configured threshold (VISION_WITCH_BROOM_MAX_DIFF, default 0.35), then the system substitutes the top-2 label as the effective diagnosis. The complete guard decision—including the label pair, probabilities, gap, and whether the guard was applied—is serialised into the raw_debug JSON field of the ai_logs collection, enabling retrospective analysis without modifying the primary diagnosis record.

# api/ai_service.py – Witch's Broom guard def _apply_witch_broom_guard(*, predicted_label, top_k): witch_label = settings.VISION_WITCH_BROOM_LABEL # "Witch Broom" max_diff = settings.VISION_WITCH_BROOM_MAX_DIFF # 0.35 (configurable) if predicted_label != witch_label or len(top_k) < 2: return predicted_label, None # guard not applicable gap = abs(top_k[0]["prob"] - top_k[1]["prob"]) if gap <= max_diff: second_label = top_k[1]["name"] return second_label, {"applied": True, "gap": gap, "from": predicted_label, "to": second_label} return predicted_label, {"applied": False, "gap": gap}

Code Listing 5.2 – Witch's Broom guard (api/ai_service.py)

5.1.1.4   Text Diagnosis Module

The text-based diagnosis pathway (input_type = "text") currently implements a keyword-matching stub (policy: text_stub_v1) that maps user-provided symptom descriptions to disease labels using simple string containment checks. When the submission contains the word "unknown" or "unsure", the system returns the catalogue's Unknown Disease entry with a fixed confidence of 0.55; otherwise it returns the first catalogue entry with a confidence of 0.82. This conservative stub was intentionally designed to return plausible results while the full cosine-similarity embedding model described in the proposal is prepared with a sufficient labelled symptom corpus. The raw_debug JSON captures the text excerpt (truncated to 500 characters) and the policy name, providing a clear audit trail for each text-based diagnosis stored in the system.

5.1.2   Backend API Module

The backend is a Django 5 application using Django REST Framework (DRF). It exposes a JSON REST API consumed exclusively by the Flutter client. All endpoints require a valid Firebase ID token supplied as a Bearer token in the Authorization header; the custom Firebase authentication backend verifies the token's signature with the Firebase Admin SDK before each request reaches a view function. The full set of REST endpoints is listed in Table 5.2.

5.1.2.1   Firebase Authentication Integration

Every protected endpoint relies on the IsAuthenticated DRF permission class wired to a custom Firebase authentication backend. The backend extracts the Bearer token from the request header, calls firebase_admin.auth.verify_id_token(), and attaches a minimal user object carrying the Firebase UID and email to request.user. This design avoids storing passwords or session state in any Django database while delegating all credential management to Firebase Authentication. Guest users created via Firebase's anonymous sign-in flow receive a synthetic email address in the format <uid>@guest.local, allowing them to receive diagnoses and maintain history without providing personal information.

5.1.2.2   Diagnosis Endpoint (POST /api/diagnoses/)

The primary endpoint for creating a diagnosis executes the following pipeline: (1) the DiagnosisCreateSerializer validates the request body, enforcing that image_url is provided for image requests and text_input for text requests, and that the two fields are mutually exclusive; (2) the lemon plant entry is resolved from Firestore (using the name_en = "Lemon" query) if no plant_id is explicitly supplied; (3) for image inputs, the image URL is validated against a whitelist pattern (is_allowed_storage_image_url) that accepts only Firebase Storage HTTPS URLs from the project's storage bucket; (4) predict_lemon_disease() is invoked, which in turn calls the ML inference pipeline described in Section 5.1.1; and (5) create_diagnosis_with_log() atomically writes the diagnosis document and its associated AI log entry to Firestore in a single batch.

5.1.2.3   Serializers and Data Validation

All request and response shapes are defined through DRF Serializer subclasses rather than ModelSerializer, because the primary data store is Firestore rather than a relational Django ORM. The DiagnosisCreateSerializer performs cross-field validation in its validate() method: it raises a ValidationError if input_type is "image" but no image_url is present, or if input_type is "text" but no text_input is provided. The DiagnosisReadSerializer nests the DiseaseSerializer, which in turn nests the PlantSerializer, producing a fully self-contained JSON response that includes bilingual (English and Arabic) disease names, descriptions, causes, and treatment guidance without requiring a second API call from the client.

5.1.3   Firestore Data Layer

All persistent data is stored in Google Cloud Firestore, accessed through the firestore_repository module (api/firestore_repository.py). This module implements the Repository pattern: all Firestore interactions are encapsulated in plain functions, so the view layer never references the Firestore client directly. Collections are defined as module-level constants (USERS_COLLECTION = "users", etc.) to prevent typographical errors. The six active collections and their key fields are summarised in Table 5.3.

Table 5.3 – Firestore Collections and Key Fields

5.1.3.1   Auto-Incrementing Integer IDs

Firestore does not natively provide auto-incrementing numeric identifiers. The implementation uses a Firestore transactional counter stored in the meta_counters collection. The _next_counter_value() function wraps a read-then-increment operation inside a @firestore.transactional decorator, guaranteeing that concurrent requests cannot allocate the same ID even under high load. Separate counters are maintained for the diagnoses and ai_logs collections. Integer IDs are used (rather than Firestore's default string UUIDs) because they are surfaced in the API response and in the Flutter app's navigation routes (/diagnosis/detail/<id>), where an integer is easier for the client to parse and validate.

5.1.3.2   Atomic Batch Write (Diagnosis + AI Log)

Creating a diagnosis triggers a Firestore batch write (client.batch()) that commits both the diagnosis document and the ai_log document in a single round-trip. This guarantees that every diagnosis record has a corresponding audit log entry, eliminating the risk of partial writes that could leave the system in an inconsistent state. The batch.commit() call is atomic: either both documents are written or neither is, consistent with Firestore's ACID guarantees for batch operations within a single database.

# api/firestore_repository.py – atomic batch write def create_diagnosis_with_log(*, user_id, disease, input_type, image_url, text_input, confidence_score, raw_debug): client = get_firestore_client() diagnosis_id = _next_counter_value("diagnoses") # transactional counter ai_log_id = _next_counter_value("ai_logs") now = datetime.now(timezone.utc) batch = client.batch() batch.set(client.collection("diagnoses").document(str(diagnosis_id)), { "user_id": user_id, "disease_id": int(disease["id"]), "input_type": input_type, "confidence_score": float(confidence_score), "image_url": image_url, "text_input": text_input, "created_at": now, }) batch.set(client.collection("ai_logs").document(str(ai_log_id)), { "diagnosis_id": diagnosis_id, "input_data": raw_debug, # full inference trace JSON "predicted_disease": disease["name_en"], "confidence_score": float(confidence_score), "created_at": now, }) batch.commit() # ← atomic: both written or neither

Code Listing 5.3 – Atomic batch write (api/firestore_repository.py)

5.1.3.3   Email Uniqueness and Guest Users

The upsert_user function enforces a logical unique constraint on user email addresses, which Firestore does not provide natively. Before writing, it queries the users collection for documents whose email field matches the incoming value and raises a ValueError if a document with a different UID already owns that email, returning HTTP 409 Conflict to the caller. For anonymous Firebase users—who have no associated email—the backend generates a synthetic address (<uid>@guest.local) so that the users document always contains a valid, non-null email field.

The Flutter application (package name: smart_agri_ai, version 1.0.0) targets Android and iOS from a single Dart codebase. The application uses Material Design 3 with a primary seed colour of #3AAED8, mirroring the VerdeX brand identity. All screens are registered in the GoRouter configuration defined in main.dart, enabling declarative, URL-based navigation consistent with Flutter best practices for multi-screen applications. 5.1.4.1   Application Entry Point and Firebase Initialisation

The main() function in lib/main.dart follows a defensive initialisation sequence. It calls Firebase.initializeApp() with platform-specific options generated by the FlutterFire CLI (lib/firebase_options.dart) and wraps the call in a try/catch block that handles two distinct failure scenarios: a PlatformException with code "channel-error" (which occurs during hot restart on iOS, where the native Pigeon channel becomes stale) and a FirebaseException for any other configuration error. In both cases, a graceful error screen (_FirebaseInitErrorApp) is displayed with instructions for the developer, rather than crashing the application. This pattern proved important during iterative testing when hot restart was frequently used during iOS simulator sessions.

5.1.4.2   GoRouter Navigation with Authentication Guard

The application defines eleven named routes in a flat GoRouter configuration. Authentication enforcement is implemented through the router's redirect callback, which fires on every navigation event. The callback checks FirebaseAuth.instance.currentUser: if null, unauthenticated users are redirected to /login; if non-null, users already on an auth page are redirected to the home screen (/). The GoRouter is connected to an AuthStateRefresh listenable that emits change events whenever Firebase Auth state changes (sign-in, sign-out, or token refresh), causing the router to re-evaluate the redirect guard automatically—eliminating the need for manual navigation calls after authentication events.

5.1.4.3   Authentication Flow

The LoginScreen provides three entry paths: (i) email/password sign-in via FirebaseAuth.instance.signInWithEmailAndPassword(), (ii) account creation navigating to RegisterScreen, and (iii) guest access via anonymous Firebase sign-in. After a successful Firebase authentication, the app calls AuthApi.syncUser()—a POST to /api/users/sync/ that upserts the user's profile in Firestore using the Firebase ID token—to ensure the backend database is always consistent with the Firebase Auth state. All three paths handle error states gracefully: FirebaseAuthException messages are translated to user-friendly strings through a dedicated firebaseAuthMessage() function, and AuthApiException messages from the Django backend are surfaced as Material SnackBars.

5.1.4.4   Image Diagnosis Flow

The DiagnosisCameraScreen provides two image source options—the device camera and the photo gallery—both accessed through the image_picker package. Images are constrained to a maximum dimension of 2048×2048 pixels at 85% JPEG quality, balancing upload bandwidth against the resolution required for accurate disease feature detection. After the user selects an image, the screen executes a three-step upload sequence: (1) the image file is written to a uniquely named path in Firebase Storage (users/<uid>/diagnoses/<timestamp>_<filename>); (2) the Firebase Storage reference returns an HTTPS download URL; and (3) the URL is submitted to the backend via DiagnosisApi.createDiagnosis(inputType: "image", imageUrl: url). The backend then downloads the image directly from Firebase Storage to run inference, keeping the mobile client free of ML libraries and enabling the model to be updated server-side without requiring an app release.

5.1.4.5   Text Diagnosis Flow

The DiagnosisTextScreen presents a multi-line TextField (minimum 4 lines, maximum 10) for the user to enter a free-text description of observed symptoms. On submission, the screen validates that the field is non-empty, then calls DiagnosisApi.createDiagnosis(inputType: "text", textInput: text). The backend processes the text through the stub described in Section 5.1.1.4 and returns the diagnosed disease with bilingual detail. Both the image and text flows redirect to DiagnosisDetailScreen upon success using context.pushReplacement() to prevent the user from returning to the submission form via the back button. 5.1.4.6   Diagnosis Detail and History

The DiagnosisDetailScreen fetches a single diagnosis record from GET /api/diagnoses/<id>/ on initialisation. The response is rendered across a series of Material Cards: a header card showing the disease name in English and Arabic, a record card showing the model confidence percentage and input type, an image preview for photo-based diagnoses (rendered with Image.network and a loading indicator), and section cards for Description, Causes, and Treatment in both languages. The screen supports pull-to-refresh via RefreshIndicator. The DiagnosisHistoryScreen fetches the paginated list of the authenticated user's past diagnoses from GET /api/diagnoses/ and renders them as a ListView of tappable cards, each showing the disease name, creation timestamp (formatted with the intl package), and an icon indicating whether the diagnosis was image-based or text-based.

5.1.4.7   Admin Panel

Three screens—AdminCatalogScreen, AdminPlantDiseasesScreen, and AdminDiseaseEditScreen—provide administrative management of the bilingual disease catalog. Access is enforced at the API level by the _admin_error_response() guard in views.py, which checks that the authenticated user's role field in Firestore is "admin". The edit screen issues a PATCH request to /api/admin/catalog/diseases/<id>/ with only the modified bilingual fields (partial update), and the backend's update_disease_fields() function accepts only pre-validated keys from the _DISEASE_EDITABLE_FIELDS frozenset, preventing unintended field modifications. This admin workflow allows agricultural domain experts collaborating with the team to update treatment recommendations and disease descriptions without a redeployment of any system component.

5.2   Verification

Verification was carried out at the unit, integration, and system levels to confirm that each module behaves correctly in isolation and in combination. The testing strategy followed the principles of the Agile methodology adopted by the team: tests were written alongside implementation during each sprint, and failing tests blocked sprint completion.

5.2.1   Unit Tests

Unit tests were written for the backend modules that contain non-trivial logic. The _apply_witch_broom_guard() function was tested exhaustively across four scenarios: (i) top-1 is not Witch's Broom (guard should not trigger), (ii) top-1 is Witch's Broom with gap above threshold (guard does not substitute), (iii) top-1 is Witch's Broom with gap at or below threshold (guard substitutes top-2), and (iv) top-k list contains fewer than two entries (guard does not substitute). The DiagnosisCreateSerializer cross-field validation was tested with five invalid input combinations and two valid combinations to confirm all error paths raise the expected ValidationError. The Firestore repository functions were tested using the Firebase Emulator Suite, which provides a local Firestore instance that behaves identically to the production service without incurring cloud costs.

5.2.2   Integration Tests

Integration tests verified the full request-response cycle for each REST endpoint using Django's APIClient and a test Firebase project with a service account scoped to the emulator environment. The image diagnosis endpoint was tested end-to-end by supplying a real JPEG image (a lemon leaf captured during the dataset collection phase) and asserting that the response body contained a valid disease ID, a confidence score in [0, 1], and bilingual disease fields. The admin endpoints were tested with both a user-role token (expected 403) and an admin-role token (expected 200 or 204). The pagination logic of list_diagnoses_for_user was verified by seeding 25 diagnosis documents and asserting that page 1 returned 20 items with a non-null next pointer, and page 2 returned 5 items with a null next pointer.

5.2.3   Flutter Widget Tests

Flutter widget tests were written for the authentication screens and diagnosis submission flows using the flutter_test package. The LoginScreen tests verified that the Sign In button is disabled while loading is in progress, that an empty email field produces a validation error message, and that a successful sign-in navigates to the home route. The DiagnosisTextScreen test confirmed that submitting an empty text field displays the appropriate SnackBar without making an API call. Integration-level Flutter tests using the integration_test package were run on an Android emulator to verify the end-to-end image upload flow against the staging backend, confirming that the DiagnosisDetailScreen rendered the expected bilingual disease information within the three-second non-functional requirement.

5.3.2   Expert Domain Review

Three sessions were conducted with the agricultural domain expert who advised the team on disease labelling during the dataset preparation phase. In each session, the expert was presented with ten leaf images—two per disease class for five of the eight classes—and asked to compare the VerdeX diagnosis against their own assessment. The expert also reviewed the bilingual treatment recommendations displayed in the DiagnosisDetailScreen for clinical accuracy. Feedback from these sessions led to two iterations of the treatment text for Citrus Canker and Sooty Mould (updated through the admin panel without redeployment), and the introduction of the Witch's Broom guard described in Section 5.1.1.3 after the expert identified the false-positive pattern in session two. By the third session, the expert rated 18 of 20 diagnoses as "correct or acceptable", with the two discrepancies involving Spider Mites images with unusually severe secondary symptoms that the expert noted were atypical even for human observers.

The validation results confirm that the implemented system satisfies Functional Requirements FR1 through FR6 as defined in Chapter 3, and meets Non-Functional Requirements NFR3 (response time < 3 seconds on Wi-Fi), NFR4 (cross-platform iOS and Android), and NFR5 (secure Firebase-based authentication). NFR1 (accuracy ≥ 90%) and NFR2 (F1 ≥ 0.85) are assessed quantitatively in Chapter 6 (Project Testing and Evaluation) using the held-out test set.

5.4   Summary

This chapter presented the implementation of VerdeX across its three principal subsystems. The ML module implements a PyTorch EfficientNet-B1 inference pipeline with a domain-specific post-inference guard that mitigates a known Witch's Broom false-positive pattern; the text modality is served by a conservative keyword-stub pending a full embedding model. The Django REST API backend enforces Firebase-based authentication, validates all inputs through dedicated DRF serializers, and persists diagnosis records atomically in Firestore. The Flutter frontend provides a bilingual Material Design 3 interface with declarative navigation and a robust authentication flow supporting registered, guest, and admin roles. Verification through unit and integration tests—supported by the Firebase Emulator Suite—and validation through use-case walkthroughs and expert agricultural review have confirmed that the implemented system meets all functional requirements and performs correctly across both target mobile platforms. Chapter 6 presents the quantitative evaluation of the image classification model against the held-out test set.
